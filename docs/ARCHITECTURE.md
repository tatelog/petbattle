# PETBATTLE アーキテクチャ

PETBATTLEは、任意のデジタル表現を戦闘可能な **Playable Expression
Token** へ変換します。ハッカソン版はラスター画像に対応し、将来SVG、
アニメーション、3D、PDF、IFCを追加してもバトルエンジンを変更しなくて
よいように、形式アダプターの境界を明確にします。

## 中心パイプライン

```text
Artifact
  -> 形式・容量検証
  -> 決定論的な画像特徴
  -> 任意のGPT-5.6 Luna意味抽出
  -> 検証済み意味エッセンス
  -> 決定論的PET Core
  -> バトルイベント列
  -> Three.jsコロシアム描画
```

モデルは意味情報の候補だけを返します。能力値や勝敗は決めません。
ルールエンジンがmanifestを検証してPET Coreへ変換し、バトルエンジンは
シード付きの純粋な状態遷移として実装します。

## 信頼境界

- ブラウザ：画像プレビュー、ローカル代替解析、3D描画、操作UI。
- 解析Worker：OpenAIの秘密鍵、スキーマ検証、意味解析キャッシュ。
- バトルルーム：行動、期限、シード付き判定、イベントログ。
- GitHub Pages：静的Webアプリと会場素材のみ。秘密情報は置きません。

## 成長ポリシー

| レベル | 対応形式 | ファイル上限 | 有効エッセンス |
| ---: | --- | ---: | ---: |
| 1 | JPEG、PNG、WebP | 2 MiB | 16 |
| 2 | ＋SVG | 5 MiB | 32 |
| 3 | ＋アニメーション・コード生成物 | 10 MiB | 64 |
| 4 | ＋GLB、OBJ、STL | 25 MiB | 128 |
| 5 | ＋PDF、IFC | 50 MiB | 256 |

ファイル容量はサービスを守るための制限で、強さにはなりません。重複せず、
作品内に根拠があり、一貫した意味エッセンスだけがPET Coreへ反映されます。

## バトル契約

`物理 > 魔法 > 防御 > 物理`。防御で物理を読んだ場合はカウンターイベントを
発生させます。双方は同一のイベント列を再生し、カメラや粒子は勝敗に影響しない
表示層として扱います。
